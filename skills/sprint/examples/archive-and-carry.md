# Example: Archive with Carry Items

> What happens when a sprint completes with incomplete features.

## Scenario

A sprint reaches the `report` phase but two of its three features have
`matchRate < 100`. The user wants to archive the completed work and start
a follow-up sprint for the leftover features.

## Final report (excerpt)

`docs/04-report/features/q2-launch.report.md` will contain a carry-items
table generated by Sprint 2 `generate-report.usecase`:

```
| Feature   | Reason         | Current Phase |
|-----------|----------------|---------------|
| billing   | matchRate 85   | iterate       |
| onboarding| s1Score 71     | qa            |
```

## Attempting archive without S4

```
/sprint archive q2-launch
```

Sprint 2 `archiveSprint` rejects the transition because the S4
`archiveReadiness` gate is false:

```
{ ok: false, reason: 'archive_readiness_fail', gateResults: { ... } }
```

## Two options for the user

### Option A: Force archive with partial completion

In Sprint 4 the skill handler propagates the gate-fail result; the user
can decide to manually transition by editing `sprint.qualityGates` or by
running `/sprint phase q2-launch --to archived` from any other phase
(transition adjacency allows `* -> archived`).

### Option B: Fork into a new sprint

(Fully wired in Sprint 5)

```
/sprint fork q2-launch --new q2-launch-followup
```

The fork creates a new sprint with only the carry-items as initial features.
The original sprint is archived as a snapshot record.

## What survives on disk after archive

```
.bkit/state/sprints/q2-launch.json     # status=archived, phase=archived
.bkit/state/sprint-status.json         # entry updated
.bkit/runtime/sprint-matrices/*.json   # final data-flow + api-contract + test-coverage
docs/04-report/features/q2-launch.report.md   # final report with carry items
```

The matrix files remain available for cross-sprint cumulative reporting in
later sprints.
